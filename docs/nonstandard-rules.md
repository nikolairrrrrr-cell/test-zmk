## Нестандартные правила и договорённости

### Тип 1 — Опоры по ОСТ 108.275.30-80 (лист / круг / метизы, лист `PDF_ZMK`)

- **Хомут**
  - В колонке `толщ-диам` всегда ставим `—`.
  - Диаметр и длину (`d24x1450`, `d30x2080` и т.п.) отражаем только в колонке `размер дет. из круга` в виде `М24х1450`, `М30х2080` и т.п.

### Тип 2 — Спецификация стали 09Г2С (лист Google **`СЕ1`**, тип PDF_ZMK2; пример 630-13)

- Колонки:
  - `отпр. элем.` — номер отправочной единицы (8, 7, 5, 4, 3, 2, 1).
  - `дет.` — номер детали внутри отправочной единицы (4, 8, 2, 1, 1, 1, 2).
  - `КОЛ.` разворачиваем в две отдельные колонки `т.` и `н.` (сейчас пустые для 630-13).
  - `МАССА В КГ` разворачиваем в три отдельные колонки `шт.`, `общ.`, `элем.`.
- Для 630-13:
  - `сечение` для метизов хранит полный текст: `Шайба 30 ГОСТ 11371-78*`, `Гайка М30.8 ГОСТ 5915-70*`.
  - Колонка `элем.` содержит только одно общее значение `102,21` в первой строке блока (c остальными строками пустыми).

### Тип 3 — Обычные спецификации `Поз / Обозначение / Наименование / Кол. / Примеч.` (лист `PDF_ZMK`, в т.ч. сложные файлы)

- **Заполнение полей (обязательно, если данные есть на чертеже):**
  - `размер дет. (лист)` ← значение из колонки `Обозначение` целиком (например, `350x200x10`, `d16x140`, `ШВ12x80` и т.п.).
  - `колич дет.` ← значение из колонки `Кол.` (число штук детали).
  - `толщ-диам` ← последняя размерная часть из `Обозначение`:
    - для размеров вида `a×b×t` → `t`;
    - для обозначений вида `d...` или `Ø...` → числовое значение диаметра;
    - для чистых метизов в текстовом виде (`Болт М8х70 ГОСТ ...`, `Гайка М8 ГОСТ ...`) поле оставляем пустым.
  - `вес 1 дет.` ← числовое значение из колонки `Примеч.` без суффикса `кг`.

- **Проверка перед записью в `pdf_zmk_baza2.csv`:**
  - Если хоть один символ в `Обозначение`, `Кол.` или массе читается с сомнением, **ячейка в CSV остаётся пустой** — никаких догадок.
  - При наличии строки `Итого` по массе проверяется сумма всех строк; при заметном расхождении данные не считаются надёжными и не заносятся, пока причина не выяснена.
  - Для повторяющихся деталей с одинаковыми `Обозначение` и `Наименованием` массы должны совпадать; расхождения трактуются как ошибка чтения и требуют повторной проверки.

- **Обязательный процесс для сложных чертежей:**
  - При запуске `pdf-zmk full` автоматически создаётся кроп таблицы (`data/recognition/<slug>.table_x4.jpg`) с апскейлом x4 через `app/recognizers/table_pipeline.py`.
  - Агент читает спецификацию **только с кропа** — исходное изображение не используется для чтения.
  - Все значения читаются минимум **в два прохода** (второй — только для проверки).
  - Если после второго прохода остаются сомнения, строка помечается к доразбору и **не попадает в JSON/Google Sheets**, пока данные не станут однозначными.
